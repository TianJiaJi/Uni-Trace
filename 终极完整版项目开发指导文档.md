# 1. 文档说明与核心原则

## 1.1 文档目标

本文档为《衣码归 · UniTrace — 校服溯源码 SaaS 平台》最终开发指导文档，整合以下全部信息：

- B 端校服企业/学校批量生成溯源码；
- C 端家长/拾获者微信扫码查询；
- 微信小程序码生成；
- 微信专用码，第三方普通扫码工具不能直接跳转业务入口；
- 小程序内部扫码；
- 家长绑定、挂失、找回；
- 家长联系方式披露授权；
- 手动输入手机号 + 短信验证码验证；
- 绑定联系电话验证；
- 隐私留痕；
- 通知家长；
- SaaS 多租户；
- 订阅配额；
- 标签导出；
- 申诉与强制解绑；
- 风控黑名单；
- 网页端订阅支付；
- 支付接口预留；
- 支付渠道配置化；
- 未配置支付参数时手动开通；
- 开源；
- 低成本单机部署；
- 多服务器扩展；
- 合规、注销、数据删除与未成年人信息保护。

## 1.2 核心原则

### 原则一：不重复造轮子

| 能力                  | 使用成熟方案                                       |
| --------------------- | -------------------------------------------------- |
| 微信专用码            | 微信官方`wxacode.getUnlimited`                   |
| 微信扫一扫            | 微信客户端原生能力                                 |
| 小程序内扫码          | `wx.scanCode`                                    |
| 微信登录              | `wx.login` + `code2Session`                    |
| 短信验证码            | 第三方短信服务                                     |
| 行为验证码/图形验证码 | 第三方验证码服务或自研轻量验证码（如 svg-captcha） |
| HTTPS                 | Caddy 自动证书                                     |
| 数据库                | PostgreSQL                                         |
| 缓存/队列             | Redis + BullMQ                                     |
| 对象存储              | MinIO / S3 兼容存储                                |
| Excel                 | exceljs                                            |
| PDF                   | pdfkit                                             |
| API 文档              | NestJS Swagger                                     |
| 限流                  | Redis + Throttler                                  |
| 监控                  | Uptime Kuma，后期 Prometheus                       |
| 网页端支付            | 微信支付 Native/H5、支付宝网页支付，接口预留       |
| 支付降级              | 未配置支付参数时线下收款 + 手动开通                |
| 二维码图形算法        | 不自研                                             |
| OCR                   | 不引入                                             |
| 虚拟号                | 当前不引入                                         |
| 邮件通知              | 后续可选，初期不引入独立邮件服务                   |

### 原则三：单机可运行，多机可扩展

```text
一台小型服务器可运行完整平台。
API 和 Worker 无状态。
后续可水平扩展。
数据库、Redis、MinIO 可逐步拆分部署。
```

---

### 原则四：微信专用码与安全边界

```text
只生成微信小程序码。
不生成普通 QR Code。
不生成 URL 二维码。
标签保留人类可读编号。
编号随机不可枚举。
小程序码 scene 随机不可枚举。
第三方普通扫码工具不能直接解析为业务入口。
```

安全边界说明：

```text
微信小程序码是微信生态专属码。
普通第三方扫码工具通常无法解析为小程序业务参数。
但不能承诺任何图像识别技术永远无法识别。
因此安全边界必须依靠：
随机编号、随机 scene、接口限流、行为验证码、登录鉴权、手机号验证、隐私留痕、黑名单与风控。
```

---

### 原则五：隐私必须留痕

```text
脱敏优先。
登录鉴权。
手机号验证。
二次确认。
完整留痕。
通知家长。
家长可关闭。
最小必要。
可申诉。
可注销。
可审计。
```

---

### 原则六：通知必须可达

```text
家长绑定、挂失、拾获、披露、紧急走丢等关键事件必须有通知机制。
优先使用微信小程序订阅消息。
重要安全事件可结合短信通知。
平台内部必须有站内通知记录。
通知内容必须脱敏。
邮件通知为后续可选渠道，初期不实现。
```

---

### 原则七：支付接口预留，配置可启用

```text
支付功能不在微信小程序内实现。
订阅付费全部在网页端完成。
系统必须预留标准支付模块。
有支付资质时，填入支付渠道配置即可启用。
未配置支付参数时，自动使用线下收款和手动开通。
支付配置不是 Cookie，而是商户号、API Key、证书、私钥、公钥等。
```

---

### 原则八：合规必须闭环

```text
用户协议
隐私政策
未成年人信息保护
个人信息收集最小化
敏感信息加密
访问留痕
数据保留期
用户注销
数据删除/匿名化
投诉申诉
安全事件处置
支付资质在生产启用前补齐
```

# 3. 合理性审计与最终取舍

## 3.1 保留的设计

| 设计              | 保留原因               |
| ----------------- | ---------------------- |
| 多租户 SaaS       | 平台化必需             |
| RBAC 权限         | 超管/租户/员工权限不同 |
| 溯源码表          | 核心资产               |
| 绑定关系表        | 家长绑定核心           |
| 挂失记录表        | 找回核心               |
| 隐私访问日志      | 合规核心               |
| 异步生成小程序码  | 微信接口有限流         |
| 异步导出          | 大批量导出不能阻塞     |
| 订阅配额          | SaaS 基础              |
| 到期后 C 端可查询 | 已印刷码不能失效       |
| 标签完整字段      | 用户明确要求           |
| 管理员强制解绑    | 任何人可绑定后必须纠错 |
| 编号输入兜底      | 小程序内扫码兼容性     |
| 家长披露授权      | 隐私可控               |
| 短信验证码        | 提高滥用成本           |
| 网页端订阅支付    | B 端付费场景正确       |
| 支付接口预留      | 后续有资质可平滑启用   |

---

## 3.2 必须调整的设计

| 原设计                                 | 最终调整                                   | 原因                                 |
| -------------------------------------- | ------------------------------------------ | ------------------------------------ |
| 支付能力笼统写"暂缓"                   | 改为小程序内不支付，网页端支付接口预留     | 订阅付费在 B 端网页端                |
| 支付资质阻塞订阅功能                   | 支付资质不阻塞开发，只阻塞生产启用         | 接口可先开发                         |
| 只支持线下开通                         | 线下开通 + 在线支付预留                    | 保证 SaaS 闭环                       |
| 支付配置写 Cookie                      | 改为支付渠道 Key/商户号/证书/私钥/公钥     | 支付配置不是 Cookie                  |
| `scene_code = Base62(trace_code.id)` | 改为随机 Base62/Base32 短码                | ID 可递增，可枚举                    |
| 绑定接口使用`traceCodeId`            | 改为`traceToken` 或 `sceneCode/codeNo` | 避免暴露内部主键                     |
| 绑定只填写联系电话                     | 绑定必须验证联系电话                       | 防止恶意填写他人手机号               |
| 普通已绑定状态可配置披露               | 默认不允许普通状态直接披露                 | 防止日常状态被滥用查看家长电话       |
| 通知家长只写流程                       | 增加通知表、订阅消息、短信兜底             | 否则通知不可落地                     |
| 申诉只有接口                           | 增加 appeal 表和处理流程                   | 申诉必须可追踪                       |
| 短信验证只有规则                       | 增加短信日志、验证码记录、图形验证码       | 防止短信盗刷和滥用                   |
| 黑名单只有概念                         | 增加 risk_blacklist 表                     | 风控必须可执行                       |
| 系统配置只有口头描述                   | 增加 system_config / tenant_config         | 限频、披露、支付、通知开关必须可配置 |
| token 命名混用 bindToken/traceToken    | 统一为 traceToken                          | 避免开发歧义                         |

---

## 3.3 暂缓的设计

| 功能                   | 暂缓原因                                   |
| ---------------------- | ------------------------------------------ |
| 微信小程序内支付       | C 端家长不付费，且小程序虚拟支付合规复杂   |
| 自动续费代扣           | 需支付渠道签约和用户授权                   |
| 开放 API               | 非 MVP                                     |
| ERP 对接               | 非 MVP                                     |
| 教育局上报             | 非 MVP                                     |
| 虚拟号/隐私号          | 成本高                                     |
| 高级审批流             | 非 MVP                                     |
| 多语言                 | 非 MVP                                     |
| 报表大屏               | 非 MVP                                     |
| Kubernetes             | 初期非必需                                 |
| Elasticsearch          | 当前不需要                                 |
| 复杂班级/年级/学籍系统 | 非 MVP                                     |
| 精确 GPS 实时定位      | 隐私风险高，非 MVP                         |
| 电子发票自动开具       | 初期记录开票信息即可                       |
| 邮件通知服务           | 初期使用站内通知 + 订阅消息 + 短信兜底即可 |

# 5. 核心业务规则

## 5.1 微信专用码规则

```text
1. 只生成微信小程序码。
2. 不生成普通 QR Code。
3. 不生成 URL 二维码。
4. 标签上显示人类可读编号。
5. 编号随机，不可顺序枚举。
6. scene_code 随机，不可使用自增 ID 转换。
7. 第三方普通扫码工具不能直接解析为业务入口。
8. 业务接口必须限流，防止编号枚举。
9. 高频查询编号必须触发验证码或临时封禁。
```

---

## 5.2 溯源码规则

每件校服一个唯一溯源码。

```text
trace_code.id          内部主键，不对外暴露
trace_code.code_no     人类可读编号，例如 UT-7K3Q-9Z2A
trace_code.scene_code  微信小程序 scene 随机短码
```

唯一性：

```text
code_no 唯一
scene_code 唯一
一个溯源码只能有一个当前有效绑定关系
```

### 5.2.1 scene_code 规则

```text
长度：8～12 位
字符集：A-Z a-z 0-9
不使用自增 ID
不使用可预测时间戳
不包含中文
不包含敏感信息
唯一索引
生成冲突时重试
```

推荐：

```text
10 位随机 Base62
```

---

### 5.2.2 code_no 规则

标签上展示：

```text
UT-7K3Q-9Z2A
```

规则：

```text
前缀 UT（UniTrace 缩写）
8 位随机字符（含校验位）
使用 Crockford Base32 字符集
排除易混淆字符：I、L、O、U
最后一位为校验位（即 7 位随机 + 1 位校验）
```

字符集：

```text
0123456789ABCDEFGHJKMNPQRSTVWXYZ
```

校验位算法推荐：

```text
使用 Luhn mod 32 或 ISO 7064 Mod 37,36 适配 Crockford Base32 字符集。
```

---

### 5.2.3 traceToken 规则

```text
详情接口返回 traceToken。
traceToken 为短期签名 token（JWT 或 HMAC 签名）。
有效期：5～10 分钟。
作用域：仅限当前溯源码的操作。
可用于：绑定、披露准备、通知家长、提交拾获线索等指定动作。
绑定成功后，该 traceToken 的绑定用途失效。
其他用途（如通知家长）在有效期内仍可使用，或重新扫码获取新 token。
traceToken 内包含 trace_code_id 的加密/签名引用，不直接暴露明文 ID。
```

---

## 5.3 绑定规则

```text
1. 任何人微信扫码后，如果校服未绑定，可以申请绑定。
2. 绑定前必须微信登录。
3. 绑定需填写学生姓名、班级、家长联系电话、关系。
4. 学校信息默认来自溯源码，不允许用户随意修改。
5. 家长联系电话必须通过短信验证码验证。
6. 未完成手机号验证，不能完成绑定。
7. 一个溯源码只能有一个 active 绑定关系。
8. 已绑定后，其他人不能继续绑定。
9. 非绑定人不可查看绑定人信息。
10. 绑定人本人可查看自己的绑定信息。
11. 绑定人可挂失、解绑。
12. 管理员可强制解绑。
13. 用户可申诉误绑定/恶意绑定。
14. 同一微信/手机号绑定数量应有上限。
15. 绑定提交必须使用 traceToken。
```

推荐限制：

| 限制项                      | 推荐值 |
| --------------------------- | -----: |
| 同一微信最多绑定            |  20 件 |
| 同一手机号最多绑定          |  20 件 |
| 同一微信每分钟绑定请求      |   3 次 |
| 同一 IP 每分钟绑定请求      |  20 次 |
| 绑定短信验证码每天每手机号  |   5 次 |
| 绑定短信验证码每天每 OpenID |   5 次 |

## 5.4 已绑定展示规则

| 扫码者     | 展示内容                                                |
| ---------- | ------------------------------------------------------- |
| 绑定人本人 | 完整校服信息、本人绑定信息、挂失/解绑/隐私设置/通知设置 |
| 非绑定人   | 校服基础信息 + "该校服已被绑定"，不展示绑定人信息       |
| 管理员     | 后台可查看脱敏信息，完整信息需权限和审计                |

普通已绑定状态下：

```text
不直接披露家长电话。
不允许直接查看家长联系方式。
可允许发送匿名紧急通知给家长。
是否允许普通紧急通知由平台配置和家长设置共同控制。
```

## 5.5 解绑规则

```text
1. 当前绑定人可主动解绑。
2. 解绑后溯源码状态恢复为未绑定。
3. 如果校服处于挂失状态，需先取消挂失，或由管理员强制处理。
4. 解绑后原绑定记录保留历史。
5. 管理员可强制解绑。
6. 强制解绑必须记录原因和日志。
7. 强制解绑应通知原绑定人。
```

---

## 5.6 挂失规则

```text
1. 只有当前绑定人可挂失自己的校服。
2. 管理员可批量挂失。
3. 挂失后状态变为 lost。
4. 挂失后可展示脱敏联系信息。
5. 失主可取消挂失。
6. 取消挂失后状态恢复为 bound。
7. 挂失和取消挂失必须记录日志。
8. 挂失状态变化应通知家长。
```

---

## 5.7 拾获规则

```text
1. 拾获者扫码后，如果校服已挂失，可查看脱敏信息。
2. 未登录不可查看完整联系方式。
3. 完整联系方式必须通过家长授权和验证流程。
4. 拾获者可提交拾获地点和留言。
5. 平台通知失主。
6. 拾获者留言不得展示给无关第三方。
7. 拾获者联系方式默认不向家长完整展示，除非拾获者同意。
```

## 5.8 家长联系方式披露授权规则

### 家长端

```text
1. 家长绑定校服成功后，系统询问是否开启"允许他人验证后联系我"。
2. 家长可选择开启、不开启、稍后设置。
3. 稍后设置默认不开启完整披露。
4. 开启披露授权前，家长联系电话必须已验证。
5. 家长可在隐私设置中随时开启或关闭。
6. 家长更换联系电话后，必须重新验证。
7. 家长更换联系电话后，披露授权自动关闭，直到重新验证。
8. 家长可查看谁申请查看过自己的联系方式。
```

### 披露者端

```text
1. 家长已开启披露授权。
2. 平台配置允许当前状态披露。
3. 校服处于可申请联系状态（见 5.9）。
4. 披露者已微信登录。
5. 披露者阅读披露提示。
6. 披露者勾选承诺。
7. 披露者手动输入本人手机号。
8. 平台发送短信验证码。
9. 披露者输入验证码。
10. 验证通过后显示家长联系电话。
11. 平台记录完整日志。
12. 平台通知家长。
```

## 5.9 可申请联系家长的状态

### 前置条件（必须同时满足）

```text
A. 家长已开启披露授权；
B. 家长联系电话已验证；
C. 平台配置允许披露；
D. 租户未被平台关闭披露功能；
E. 披露者未命中黑名单且未超过限频。
```

### 允许披露的状态条件（满足其一）

```text
1. 校服已挂失（status = lost）；
2. 家长标记孩子走丢/紧急联系（emergency_missing_report 处于 active）；
3. 家长主动开启临时紧急联系。
```

### 默认策略

```text
普通已绑定未挂失（status = bound 且无紧急状态）：
  不直接披露家长电话。
  只允许发送紧急通知给家长（是否允许由平台配置和家长设置控制）。

已挂失/走丢/紧急状态：
  满足全部前置条件后，允许验证后查看家长电话。
```

## 5.10 孩子走丢/紧急状态

家长可在小程序中标记：

```text
孩子走丢 / 紧急联系
```

```text
该溯源码进入紧急联系可用状态；
帮助者可申请验证后联系家长；
平台向家长发送确认通知（确认紧急状态已开启）；
可设置有效期，默认 24 小时；
可取消；
可续期；
到期自动关闭；
续期次数默认最多 3 次；
超过次数需重新创建。
```

## 5.11 订阅到期规则

```text
1. 租户订阅到期后，C 端扫码查询继续可用。
2. 已生成溯源码不因租户到期而失效。
3. 租户到期后，B 端禁止新增、生成、导出、批量操作。
4. 租户到期后一定时间内可只读和续费。

6. 租户被平台风控暂停时，C 端基础查询可保留，但披露功能可由平台关闭。
```

推荐宽限期：

```text
到期后 30 天内只读。
超过 30 天仍保留 C 端查询，但 B 端后台可限制登录或仅超管可访问。
```

## 5.16 注销与删除规则

```text
1. C 端用户可申请注销账号。
2. 注销前必须确认当前账号无未处理申诉或安全风险。
3. 如果用户有活跃绑定关系，必须先解绑或由管理员处理。
4. 用户注销后，个人信息匿名化。
5. 绑定历史、披露日志、隐私访问日志按合规要求保留。
6. 保留期结束后删除或不可逆匿名化。
7. 用户可请求删除或更正其提供的个人信息，法律要求保留的除外。
```

注销前绑定处理规则：

```text
用户发起注销时，系统检查是否存在 active 状态的绑定。
如果存在，提示用户先解绑所有校服，或联系管理员处理。
用户确认解绑后，绑定状态变为 released，release_reason = 'account_deleted'。
解绑完成后继续注销流程。
```

```text
有支付资质时，填入对应支付渠道配置即可启用。
未配置支付参数时，系统自动降级为线下收款 + 手动开通。
```

支付渠道配置不是 Cookie，而是：

```text
商户号
API Key
API v3 Key
证书序列号
商户私钥
支付宝应用私钥
支付宝公钥
回调地址
支付目录
沙箱/生产环境标识
```

### 5.18.6 线上支付成功后的动作

```text
1. 更新订单状态为 paid；
2. 记录 payment；
3. 创建或续期 subscription；
4. 增加码量配额；
5. 写入 subscription_change_log；
6. 通知租户管理员；
7. 生成站内通知；
8. 如需要，触发发票流程。
```

---

### 5.18.7 线下收款后的动作

```text
1. 租户提交付款凭证或超管创建线下订单；
2. 超管确认到账；
3. 手动标记订单 paid；
4. 手动或自动开通订阅；
5. 增加码量配额；
6. 通知租户。
```

---

### 5.18.8 订阅到期与续费

```text
1. 订阅到期前 15 天、7 天、3 天、1 天提醒（BullMQ 定时任务）。
2. 到期后进入宽限期，默认 30 天。
3. 宽限期内 B 端只读，不能新增、生成、导出。
4. 宽限期内可续费。
5. 续费成功后恢复可用。
6. C 端已生成码查询不受租户到期影响。
7. 超管可手动暂停/恢复租户。
```

### 5.18.9 升级/降级规则

MVP 推荐：

```text
升级：立即生效，按剩余周期补差价或人工处理。
降级：当前周期不退款，下一周期生效。
码量包：购买后立即增加配额。
```

如果初期不想做复杂计价：

```text
升级/降级全部由超管人工处理。
网页端只提交申请，不自动计算差价。
```

---

### 5.18.10 退款规则

```text
1. 未开通且未使用码量，可申请退款。
2. 已开通但码量未使用，可人工审核退款。
3. 已生成溯源码或已使用码量，默认不退。
4. 退款必须创建 refund 记录。
5. 退款成功后可扣减配额或暂停订阅。
6. 退款需超管或财务审批。
7. 线上退款通过支付渠道退款接口执行。
8. 线下退款人工转账并记录凭证。
```

# 6. 核心业务流程

## 6.1 B 端主流程

```text
访问官网（unitrace.cn）
  → 注册/申请试用
  → 平台审核或自动试用
  → 选择套餐
  → 网页端支付或线下付款
  → 支付成功自动开通，或超管手动开通
  → 登录租户后台
  → 创建学校/款式/批次
  → 批量创建溯源码
  → 生成微信小程序码
  → 导出 Excel/PDF 标签
  → 印刷/缝制
  → 发放给学生
  → 家长扫码绑定
  → 数据统计与后续管理
```

---

## 6.2 批量生成溯源码流程

```text
租户员工创建批次
  → 选择学校、款式、尺码、数量
  → 校验订阅状态和剩余码量
  → Redis 预扣码量
  → 创建 generation_task
  → 生成 trace_code 记录（status = pending_qr）
  → 推入 BullMQ 队列
  → Worker 调用微信 wxacode.getUnlimited
  → 图片上传 MinIO
  → 写入 wxacode_asset
  → 更新 generation_task 成功/失败数量
  → trace_code.status = unbound
  → 失败进入重试队列或死信队列
```

前置条件：

```text
只有 status = pending_qr 的溯源码可提交生成。
生成成功后状态变为 unbound。
生成失败后状态保持 pending_qr，可重试。
```

---

## 6.3 C 端扫码总流程

```text
用户微信扫码
  → 微信识别小程序码
  → 打开小程序「衣码归」pages/trace/detail
  → 解析 scene
  → 请求后端详情接口（无需登录即可获取基础信息）
  → 后端判断码状态和访问者身份
  → 返回差异化数据
  → 返回 traceToken
  → 小程序展示页面
```

说明：

```text
详情接口的基础信息（学校、款式、尺码、状态等）无需登录即可访问。
敏感操作（绑定、披露、通知等）需要登录和 traceToken。
未登录用户通过 IP + 设备指纹进行限流。
已登录用户通过 OpenID 进行限流。
```

---

## 6.4 小程序内扫码流程

```text
用户进入小程序「衣码归」
  → 点击"扫一扫"
  → 调用 wx.scanCode
  → 扫描校服上的微信小程序码
  → 解析扫码结果
      1. 从 path 中解析 scene
      2. 从 result 中解析 scene_code
      3. 从 result 中解析 code_no
  → 校验 code_no 校验位
  → 跳转详情页
  → 如果解析失败
      → 提示手动输入编号
      → 手动输入触发限流和验证码
```

---

## 6.5 绑定流程

```text
扫码
  → 状态 unbound
  → 点击立即绑定
  → 微信登录
  → 使用详情接口返回的 traceToken
  → 填写学生姓名、班级、关系
  → 输入家长联系电话
  → 发送绑定短信验证码
  → 输入短信验证码
  → 勾选隐私协议
  → 请求订阅消息授权
  → 提交绑定
  → 后端事务校验
  → 校验 traceToken 有效性和作用域
  → 校验手机号验证码
  → 插入 binding
  → contact_phone_verified = true
  → 更新 trace_code.status = bound
  → 绑定成功
  → 询问是否开启家长联系方式披露授权
```

---

## 6.6 挂失流程

```text
家长进入我的校服
  → 点击挂失
  → 填写遗失原因/地点
  → 确认挂失
  → 创建 lost_report
  → trace_code.status = lost
  → 通知家长
```

---

## 6.7 家长联系方式披露流程

```text
帮助者扫码
  → 进入详情页
  → 点击"申请联系家长"
  → 平台判断当前状态是否允许披露（挂失/走丢/紧急）
  → 平台判断家长是否开启披露授权
  → 平台判断家长联系电话是否已验证
  → 如果未开启或不允许，只允许发送紧急通知
  → 如果已开启且允许，进入披露流程
  → 微信登录
  → 弹出披露提示
  → 勾选承诺
  → 输入本人手机号
  → 图形验证码/滑块
  → 发送短信验证码
  → 输入验证码
  → 验证通过
  → 显示家长联系电话（一次性展示）
  → 平台记录日志
  → 平台通知家长
```

---

## 6.8 管理员强制解绑流程

```text
家长/学校发现误绑定
  → 提交申诉或联系管理员
  → 管理员根据编号查询
  → 强制解绑
  → 填写原因
  → binding.status = released
  → trace_code.status = unbound
  → trace_code.current_binding_id = NULL
  → 记录操作日志
  → 通知原绑定人
```

---

## 6.9 订阅开通与续费流程

### 6.9.1 线下收款流程

```text
客户申请试用/联系销售
  → 平台确认套餐和码量
  → 客户对公转账/合同签约
  → 超管创建线下订单
  → 客户或超管上传付款凭证
  → 超管确认到账
  → 订单状态 paid
  → 手动开通订阅
  → 增加码量配额
  → 通知租户
```

---

### 6.9.2 网页端在线支付流程

```text
租户管理员登录网页后台
  → 进入订阅中心
  → 选择套餐/码量包/续费周期
  → 创建订单
  → 进入收银台
  → 选择微信支付/支付宝/对公转账
  → 扫码支付或跳转支付
  → 支付渠道回调平台
  → 平台验签
  → 幂等处理
  → 订单状态 paid
  → 自动开通/续期订阅
  → 增加配额
  → 通知租户
  → 返回支付成功页
```

---

### 6.9.3 支付未配置时的降级流程

```text
系统检测支付渠道未启用
  → 收银台只展示对公转账/线下付款
  → 租户上传付款凭证
  → 超管后台确认到账
  → 手动标记订单已支付
  → 手动开通订阅
```

---

### 6.9.4 支付掉单补偿流程

```text
用户支付成功但回调失败
  → 前端轮询订单状态（每 3 秒，最多 60 秒）
  → 后端主动调用支付渠道查询订单
  → 若渠道已支付，则本地补单
  → 更新订单 paid
  → 开通订阅
  → 记录异常补偿日志
```

定时补偿任务：

```text
BullMQ 定时任务每 5 分钟扫描 status = paying 且超过 5 分钟的订单。
主动查询支付渠道状态。
如果渠道已支付则补单。
如果渠道未支付且超过订单有效期则标记 expired。
```

---

## 6.10 通知流程

```text
业务事件发生
  → 写入 notification 记录
  → 判断用户订阅消息授权
  → 如果可发送订阅消息，则调用微信订阅消息
  → 如果为高优先级安全事件且租户/平台允许短信兜底，则发送短信
  → 更新通知状态
  → 用户进入通知中心可查看
```

---

## 6.11 申诉流程

```text
用户提交申诉
  → 创建 appeal
  → 状态 pending
  → 客服/管理员处理
  → 可强制解绑、关闭披露、拉黑、驳回
  → 填写处理结果
  → appeal.status = resolved/rejected
  → 通知申诉人
```

---

## 6.12 账号注销流程

```text
用户进入个人中心
  → 点击注销账号
  → 阅读注销提示
  → 短信验证或微信验证
  → 检查未处理申诉/安全风险
  → 检查是否存在活跃绑定
      → 如有，提示先解绑或联系管理员
      → 用户确认解绑所有绑定
  → 用户状态 deleted
  → 个人信息匿名化
  → 保留审计日志
```

# 7. 状态机设计

```text

    │
    │ 提交生成小程序码
    ▼

    │
    │ 生成成功
    ▼
unbound
    │
    │ 家长绑定
    ▼

    │

    ▼

    │
```

## 9.4 小程序码生成流程


## 9.8 解析扫码结果

```ts
type ScanParseResult =
  | { type: 'scene'; value: string }
  | { type: 'codeNo'; value: string };

function parseScanResult(res: any): ScanParseResult | null {
  // 微信小程序码扫码结果中 path 包含 scene 参数
  if (res.path) {
    const match = res.path.match(/[?&]scene=([^&]+)/);
    if (match && match[1]) {
      return {
        type: 'scene',
        value: decodeURIComponent(match[1])
      };
    }
  }

  if (res.result) {
    const result = decodeURIComponent(res.result).trim();

    // 匹配 scene_code：8-12 位字母数字
    if (/^[A-Za-z0-9]{8,12}$/.test(result)) {
      return {
        type: 'scene',
        value: result
      };
    }

    // 匹配 code_no：UT-XXXX-XXXX 格式
    if (/^UT-[0-9A-HJKMNP-TV-Z]{4}-[0-9A-HJKMNP-TV-Z]{4}$/i.test(result)) {
      const codeNo = result.toUpperCase();
      if (validateCodeNoChecksum(codeNo)) {
        return {
          type: 'codeNo',
          value: codeNo
        };
      }
    }
  }

  return null;
}
```

# 10. 家长联系方式披露功能

## 10.1 功能定义

```text
家长可以选择是否允许他人在紧急或必要情况下，
通过身份验证后查看自己的联系电话。
```

---

## 10.2 家长授权规则

```text
1. 家长绑定校服成功后，系统询问是否开启。
2. 家长可选择开启、不开启、稍后设置。
3. 稍后设置默认不开启完整披露。
4. 开启前家长联系电话必须已验证。
5. 家长后续可在设置中修改。
6. 家长关闭后，他人不能查看完整家长电话。
7. 家长可查看披露记录。
8. 家长更换手机号后，披露授权自动关闭。
```

---

## 10.3 披露者验证规则

```text
1. 披露者必须微信登录。
2. 披露者必须点击申请联系家长。
3. 披露者必须阅读披露提示。
4. 披露者必须勾选承诺。
5. 披露者手动输入本人手机号。
6. 平台发送短信验证码。
7. 披露者输入验证码。
8. 验证通过后才显示家长联系电话。
9. 平台记录完整日志。
10. 平台通知家长。
```

---

## 10.4 披露前置条件

```text
1. 溯源码未作废；
2. 租户未被平台关闭披露；
3. 家长已开启披露授权；
4. 家长联系电话已验证；
5. 当前状态允许披露（挂失/走丢/紧急）；
6. 披露者未命中黑名单；
7. 披露者未超过限频；
8. 图形验证码/滑块通过；
9. 短信验证码验证通过。
```

---

## 10.5 披露内容范围

建议只显示：

```text
家长称呼：张家长
家长电话：13800005678
学校名称：XX小学
校服款式：春秋运动服
尺码：140
```

不显示：

```text
学生完整姓名；
班级；
年级；
家长完整姓名；
家庭住址；
其他隐私信息。
```

---

## 10.6 披露有效期与展示规则

```text
短信验证码有效期：5 分钟。
验证通过后，家长电话仅展示一次。
展示页面有效期：10 分钟（超时页面自动关闭，需重新验证）。
前端不长期缓存家长电话。
后端返回一次性展示数据，不返回可重复使用的 revealToken。
每次查看均记录为独立的隐私访问事件。
如果用户关闭页面后再次申请，需重新走完整验证流程。
```

## 14.3 鉴权方式

```text
B 端：账号密码登录 + JWT access_token + refresh_token
C 端：微信登录 + JWT access_token
敏感操作：短期 traceToken
支付回调：签名验证，不依赖用户登录态
```

接口访问级别：

```text
公开接口：扫码详情基础信息（无需登录，IP 限流）
C 端登录接口：微信登录（无需 token）
C 端认证接口：绑定、挂失、披露等（需 JWT + traceToken）
B 端认证接口：后台管理（需 JWT）
超管接口：平台管理（需超管 JWT）
回调接口：支付回调（签名验证，无需 JWT）
```

建议：

```text
access_token 有效期：2 小时
refresh_token 有效期：14 天
refresh_token 轮换
C 端敏感操作要求重新验证或短信验证
```


处理规则：

```text
1. 验签。
2. 校验商户号。
3. 校验订单金额。
4. 幂等处理（同一 transaction_id 不重复处理）。
5. 更新 payment。
6. 更新 order。
7. 开通订阅。
8. 返回成功应答。
```

安全要求：

```text

回调接口通过签名验证确认来源。
微信支付回调来源 IP 可选白名单校验。
回调接口必须防重放（时间戳 + nonce）。
```

# 16. 安全、隐私与合规

## 16.1 数据分类

| 数据         | 敏感级别 | 处理方式                   |
| ------------ | -------- | -------------------------- |
| 学生姓名     | 高       | 加密存储，展示脱敏         |
| 家长手机号   | 高       | 加密存储，展示脱敏         |
| 披露者手机号 | 高       | 加密存储，展示脱敏         |
| 班级         | 中       | 按需展示                   |
| 微信 OpenID  | 中       | 身份识别                   |
| 校服基础信息 | 低       | 可公开                     |
| 隐私访问日志 | 高       | 长期保存                   |
| 申诉附件     | 高       | 私有桶签名 URL             |
| 支付凭证     | 高       | 私有桶签名 URL             |
| 支付密钥     | 极高     | 环境变量/KMS，禁止入库明文 |

## 16.2 脱敏规则

```text
张三 → 张*
张小明 → 张*明
13800005678 → 138****5678
```

## 16.3 加密规则

| 数据       | 加密方式                                 |
| ---------- | ---------------------------------------- |
| 密码       | argon2id 或 bcrypt                       |
| 手机号     | AES-256-GCM                              |
| 学生姓名   | AES-256-GCM                              |
| 文件 URL   | 临时签名                                 |
| 数据库备份 | 加密存储                                 |
| phone_hash | HMAC-SHA256 + pepper                     |
| 短信验证码 | Redis 短期存储，日志只存哈希或脱敏       |
| 支付密钥   | 环境变量/KMS/配置中心                    |
| traceToken | HMAC-SHA256 签名，内含加密 trace_code_id |

## 16.4 密钥管理

```text
AES 主密钥不得写入代码仓库。
支付商户密钥不得写入代码仓库。
生产环境使用环境变量或 KMS。
支持密钥轮换。
不同环境使用不同密钥。
备份文件加密。
导出文件签名 URL 有效期不超过 1 小时。
支付私钥文件权限严格控制（chmod 600）。
```

# 24. 最终开发基线总结

```text
1. 项目名称：衣码归（UniTrace）。
2. 项目核心是不重复造轮子。
3. 使用微信小程序码，不生成普通二维码。
4. 第三方普通扫码工具不能直接解析为业务入口，但不承诺绝对不可识别。
5. 标签包含小程序码、编号（UT 前缀）、学校、款式、尺码、颜色、批次、生产日期。
6. 小程序「衣码归」内提供扫一扫，使用 wx.scanCode。
7. 小程序内扫码失败时提供编号输入兜底。
8. code_no 必须随机并带校验位（Crockford Base32 + Luhn mod 32），前缀 UT。
9. scene_code 必须随机，不能使用 Base62(trace_code.id)。
10. 绑定接口不直接暴露 trace_code.id，使用 traceToken。
11. traceToken 为短期签名 token，有效期 5～10 分钟，绑定成功后绑定用途失效。
12. 任何人可绑定，但一件校服只能绑定一次。
13. 绑定必须验证家长联系电话。
14. 已绑定后其他人不能继续绑定。
15. 非绑定人不可查看绑定人信息。
16. 挂失后拾获者先看到脱敏信息。
17. 家长联系方式披露功能由家长自主开启或关闭。
18. 开启披露授权前家长手机号必须已验证。
19. 绑定校服时询问家长是否开启披露授权。
20. 家长可在设置中随时修改。
21. 家长更换手机号后披露授权自动关闭。
22. 其他人申请查看家长电话必须微信登录。
23. 其他人必须手动输入本人手机号。
24. 其他人必须通过短信验证码验证。
25. 验证通过并确认承诺后才显示家长电话（一次性展示）。
26. 平台必须记录披露者 OpenID、手机号、IP、设备、时间、结果。
27. 平台必须通知家长。
28. 普通已绑定状态默认不直接披露家长电话。
29. 披露仅在挂失/走丢/紧急状态下允许（需同时满足前置条件）。
30. 不做微信获取手机号快速验证。
31. 订阅到期后 C 端扫码查询继续有效。
32. 租户到期后 B 端禁止新增、生成、导出、批量操作。
33. 当前无小程序资质，开发可先进行。
34. 正式印刷前必须完成正式小程序认证。
35. 正式印刷必须使用 release 版小程序码。
36. 微信小程序内不接入支付。
37. 订阅付费全部在网页端完成。
38. 支付接口必须预留。
39. 有支付资质时，填入支付渠道 Key/商户号/证书即可启用。
40. 未配置支付参数时，使用线下收款和手动开通。
41. 支付配置不是 Cookie，而是商户密钥、API Key、证书、私钥、公钥等。
42. 线上支付成功后自动开通订阅。
43. 线下收款由超管确认后手动开通。
44. 所有订单、支付、退款、订阅变更必须留痕。
45. 技术栈统一为 React + NestJS + TypeScript + PostgreSQL + Redis + MinIO。
46. 小程序使用原生微信小程序 + TypeScript。
47. 初期不拆分开源版和商业版，所有核心功能统一实现。
48. 开源许可证发布前最终确认，若防止闭源商用推荐 AGPL-3.0。
49. 单机 Docker Compose 可运行。
50. API/Worker 无状态，后续可扩展到多台服务器。
51. 必须提供申诉、通知、黑名单、注销、数据删除/匿名化和审计能力。
52. 未登录用户通过 IP + 设备指纹限流，已登录用户通过 OpenID 限流。
53. 定时任务使用 BullMQ repeatable jobs 实现。
54. 账号注销前必须处理活跃绑定。
55. 邮件通知为后续可选，初期不实现。
56. GitHub 仓库名：unitrace。npm 包前缀：@unitrace/。数据库名：unitrace。
```

## 25.2 配置说明

```text
PAYMENT_ENABLED=false 时，所有在线支付不可用。
PAYMENT_OFFLINE_ENABLED=true 时，线下收款和手动开通可用。
有支付资质后，将对应渠道配置填入，并开启对应开关。
不填支付参数时，系统不报错，只展示线下收款。
支付参数填写后，系统自动启用对应支付渠道。
```

## 25.3 最终执行口径

```text
支付功能先做接口和流程。
有没有资质不影响开发。
有资质就填配置启用。
没资质就线下收款、手动开通。
后续拿到资质后，只需要补配置，不需要重构。
```
