# 1. 文档说明与核心原则

## 1.1 文档目标

本文档为《衣码归 · UniTrace — 校服溯源码 SaaS 平台》开发指导文档，整合以下信息：

- B 端校服企业/学校批量生成溯源码；
- C 端家长/拾获者微信扫码查询；
- 微信小程序码生成；
- 微信专用码，第三方普通扫码工具不能直接跳转业务入口；
- 小程序内部扫码；
- 家长绑定（微信登录即可，无需电话、无需短信）；
- 挂失（纯展示状态：扫码时强调显示「已丢失」+ 学校 + 班级 + 归还指引）；
- SaaS 多租户；
- 订阅配额；
- 标签导出；
- 申诉与强制解绑；
- 风控黑名单；
- 网页端订阅支付；
- 支付接口预留；
- 支付渠道配置化；
- 未配置支付参数时手动开通；
- 开源；
- 低成本单机部署；
- 多服务器扩展；
- 合规、注销、数据删除。

> v5.0 已明确砍除的能力（不再实现，不在范围内）：
> 家长联系方式披露、紧急走丢/孩子走丢状态、短信验证码、图形/行为验证码、C 端通知家长（订阅消息/短信）、拾获者留言与拾获者电话、学生姓名与家长电话存储。
> 砍除理由见第 3 章。

## 1.2 核心原则

### 原则一：不重复造轮子

| 能力            | 使用成熟方案                       |
| --------------- | ---------------------------------- |
| 微信专用码      | 微信官方 `wxacode.getUnlimited`    |
| 微信扫一扫      | 微信客户端原生能力                 |
| 小程序内扫码    | `wx.scanCode`                      |
| 微信登录        | `wx.login` + `code2Session`        |
| HTTPS           | Caddy 自动证书                     |
| 数据库          | PostgreSQL                         |
| 缓存/队列       | Redis + BullMQ                     |
| 对象存储        | MinIO / S3 兼容存储                |
| Excel           | exceljs                            |
| PDF             | pdfkit                             |
| API 文档        | NestJS Swagger                     |
| 限流            | Redis + Throttler                  |
| 监控            | Uptime Kuma，后期 Prometheus       |
| 网页端支付      | 微信支付 Native/H5、支付宝网页支付，接口预留 |
| 支付降级        | 未配置支付参数时线下收款 + 手动开通 |
| 二维码图形算法  | 不自研                             |
| OCR             | 不引入                             |
| 虚拟号          | 不引入                             |
| 短信服务        | **不引入**（v5.0 砍除）            |
| 行为/图形验证码 | **不引入**（v5.0 砍除）            |

### 原则二：先实现核心功能

所有核心功能统一实现：

```text

标签导出

合规与注销

```

### 原则三：单机可运行，多机可扩展

```text
一台小型服务器可运行完整平台。
API 和 Worker 无状态。
后续可水平扩展。
数据库、Redis、MinIO 可逐步拆分部署。
```

---

### 原则四：微信专用码与安全边界

```text
只生成微信小程序码。
不生成普通 QR Code。
不生成 URL 二维码。
标签保留人类可读编号。
编号随机不可枚举。
小程序码 scene 随机不可枚举。
第三方普通扫码工具不能直接解析为业务入口。
```

安全边界说明：

```text
微信小程序码是微信生态专属码。
普通第三方扫码工具通常无法解析为小程序业务参数。
但不能承诺任何图像识别技术永远无法识别。
因此安全边界依靠：
随机编号、随机 scene、接口限流、登录鉴权、黑名单与风控。
```

---

### 原则五：最小信息原则（v5.0 重写）

```text
系统只保存学校与班级两类可定位到集体的信息。
不保存学生姓名、家长姓名、家长电话、家庭住址等个人身份信息。
因此不存在「个人联系方式泄露」风险。
不需要披露验证、隐私访问留痕、二次确认等复杂机制。
未成年人个人信息保护压力大幅降低。
绑定人身份仅用微信 OpenID（假名标识）做权限判断，不视为个人敏感信息。
```

---

### 原则六：支付接口预留，配置可启用

```text
支付功能不在微信小程序内实现。
订阅付费全部在网页端完成。
系统必须预留标准支付模块。
有支付资质时，填入支付渠道配置即可启用。
未配置支付参数时，自动使用线下收款和手动开通。
支付配置不是 Cookie，而是商户号、API Key、证书、私钥、公钥等。
```

---

### 原则七：合规必须闭环

```text
用户协议
隐私政策
个人信息收集最小化（只存学校与班级）
访问留痕（操作日志）
数据保留期
用户注销
数据删除/匿名化
投诉申诉
支付资质在生产启用前补齐
```

# 3. 合理性审计与最终取舍

## 3.1 v5.0 砍除的设计及理由

| 砍除的设计                 | 砍除理由                                                                 |
| -------------------------- | ------------------------------------------------------------------------ |
| 家长联系方式披露           | 伪需求：人走丢会报警，没人看校服标签找电话；日常拾获应送回学校而非打家长电话 |
| 紧急走丢/孩子走丢状态      | 同上，报警是正确路径；平台不应承担寻人职能                               |
| 短信验证码                 | 不再验证家长电话，短信无使用场景；省掉服务商、签名审核、成本与防刷        |
| 图形/行为验证码            | 不再有短信盗刷场景；编号枚举靠限流 + 随机码 + 校验位即可                  |
| C 端通知家长（订阅消息/短信） | 家长是挂失发起方无需被通知；不引入通知渠道降低复杂度                     |
| 拾获者留言 / 拾获者电话    | 拾获只需送回学校班级，不需要留言和拾获者联系方式                         |
| 学生姓名 / 家长电话存储    | 只存学校与班级即可定位到集体，不存个人身份信息                           |
| 隐私访问日志 / 披露日志    | 不存在个人联系方式披露，无隐私访问需留痕                                 |

> 砍除后的核心闭环：建码 → 扫码 → 绑定（微信登录+班级）→ 挂失（纯展示）→ 送回学校班级。
> 学校外丢失不在产品责任范围内（用户明确：关我屁事）。

---

## 3.2 保留的设计

| 设计              | 保留原因               |
| ----------------- | ---------------------- |
| 多租户 SaaS       | 平台化必需             |
| RBAC 权限         | 超管/租户/员工权限不同 |
| 溯源码表          | 核心资产               |
| 绑定关系表        | 家长绑定核心           |
| 挂失状态          | 找回核心（纯展示）     |
| 异步生成小程序码  | 微信接口有限流         |
| 异步导出          | 大批量导出不能阻塞     |
| 订阅配额          | SaaS 基础              |
| 到期后 C 端可查询 | 已印刷码不能失效       |
| 标签完整字段      | 用户明确要求           |
| 管理员强制解绑    | 任何人可绑定后必须纠错 |
| 编号输入兜底      | 小程序内扫码兼容性     |
| 网页端订阅支付    | B 端付费场景正确       |
| 支付接口预留      | 后续有资质可平滑启用   |
| 绑定功能租户开关  | 降级为通用电子标签，提升通用性 |

---

## 3.3 暂缓的设计

| 功能                   | 暂缓原因                                   |
| ---------------------- | ------------------------------------------ |
| 微信小程序内支付       | C 端家长不付费，且小程序虚拟支付合规复杂   |
| 自动续费代扣           | 需支付渠道签约和用户授权                   |
| 开放 API               | 非 MVP                                     |
| ERP 对接               | 非 MVP                                     |
| 教育局上报             | 非 MVP                                     |
| 虚拟号/隐私号          | 成本高                                     |
| 高级审批流             | 非 MVP                                     |
| 多语言                 | 非 MVP                                     |
| 报表大屏               | 非 MVP                                     |
| Kubernetes             | 初期非必需                                 |
| Elasticsearch          | 当前不需要                                 |
| 复杂班级/年级/学籍系统 | 非 MVP                                     |
| 电子发票自动开具       | 初期记录开票信息即可                       |
| 邮件通知服务           | 初期使用站内通知即可                       |

# 5. 核心业务规则

## 5.1 微信专用码规则

```text
1. 只生成微信小程序码。
2. 不生成普通 QR Code。
3. 不生成 URL 二维码。
4. 标签上显示人类可读编号。
5. 编号随机，不可顺序枚举。
6. scene_code 随机，不可使用自增 ID 转换。
7. 第三方普通扫码工具不能直接解析为业务入口。
8. 业务接口必须限流，防止编号枚举。
9. 高频查询编号触发临时限流或封禁。
```

---

## 5.2 溯源码规则

每件校服一个唯一溯源码。

```text
trace_code.id          内部主键，不对外暴露
trace_code.code_no     人类可读编号，例如 UT-7K3Q-9Z2A
trace_code.scene_code  微信小程序 scene 随机短码
```

唯一性：

```text
code_no 唯一
scene_code 唯一
一个溯源码只能有一个当前有效绑定关系
```

### 5.2.1 scene_code 规则

```text
长度：8～12 位
字符集：A-Z a-z 0-9
不使用自增 ID
不使用可预测时间戳
不包含中文
不包含敏感信息
唯一索引
生成冲突时重试
```

推荐：

```text
10 位随机 Base62
```

---

### 5.2.2 code_no 规则

标签上展示：

```text
UT-7K3Q-9Z2A
```

规则：

```text
前缀 UT（UniTrace 缩写）
8 位随机字符（含校验位）
使用 Crockford Base32 字符集
排除易混淆字符：I、L、O、U
最后一位为校验位（即 7 位随机 + 1 位校验）
```

字符集：

```text
0123456789ABCDEFGHJKMNPQRSTVWXYZ
```

校验位算法推荐：

```text
使用 Luhn mod 32 或 ISO 7064 Mod 37,36 适配 Crockford Base32 字符集。
```

---

### 5.2.3 traceToken 规则

```text
详情接口返回 traceToken。
traceToken 为短期签名 token（JWT 或 HMAC 签名）。
有效期：5～10 分钟。
作用域：仅限当前溯源码的操作。
可用于：绑定等指定动作。
绑定成功后，该 traceToken 的绑定用途失效。
traceToken 内包含 trace_code_id 的加密/签名引用，不直接暴露明文 ID。
```

---

## 5.3 绑定规则

```text
1. 绑定功能受租户开关控制（tenant_config.binding.enabled）。
2. 租户开启绑定时：未绑定校服可被扫码绑定。
3. 租户关闭绑定时：扫码只展示物品信息，无登录、无绑定入口（普通电子标签）。
4. 绑定前必须微信登录（wx.login + code2Session）。
5. 绑定只需填写「班级」一项信息（学校来自溯源码，不可修改）。
6. 不收集学生姓名、家长电话、关系等个人身份信息。
7. 一个溯源码只能有一个 active 绑定关系。
8. 已绑定后，其他人不能继续绑定。
9. 绑定人本人可查看自己的绑定、挂失、解绑。
10. 非绑定人扫码不展示绑定人任何信息。
11. 管理员可强制解绑。
12. 用户可申诉误绑定/恶意绑定。
13. 同一微信绑定数量应有上限。
14. 绑定提交必须使用 traceToken。
```

推荐限制：

| 限制项                      | 推荐值 |
| --------------------------- | -----: |
| 同一微信最多绑定            |  20 件 |
| 同一微信每分钟绑定请求      |   3 次 |
| 同一 IP 每分钟绑定请求      |  20 次 |

> 因为不存个人敏感信息，恶意绑定的危害仅为「占位导致真主人无法绑定」，可通过强制解绑 + 申诉低成本纠正，故不再需要电话验证或短信验证。

## 5.5 解绑规则

```text
1. 当前绑定人可主动解绑。
2. 解绑后溯源码状态恢复为未绑定。
3. 如果校服处于挂失状态，需先取消挂失，或由管理员强制处理。
4. 解绑后原绑定记录保留历史。
5. 管理员可强制解绑。
6. 强制解绑必须记录原因和日志。
```

## 5.6 挂失规则（v5.0：纯展示状态）

```text
1. 只有当前绑定人可挂失自己的校服。
2. 管理员可批量挂失。
3. 挂失后 trace_code.status = lost。
4. 挂失后任何人扫码都会看到醒目的「该衣物已丢失」红色状态。
5. 挂失扫码页同时展示：学校 + 班级 + 归还指引（如拾获请送至 XX学校 X班级）。
6. 失主可取消挂失，状态恢复为 bound。
7. 挂失和取消挂失必须记录日志。
8. 不通知家长（家长是发起方）。
9. 不涉及任何联系方式披露或验证。
```

> 挂失按钮的唯一作用：让扫码者一眼看到「已丢失」并知道往哪归还。学校内丢失送对应班级，学校外丢失不在产品责任范围内。

# 6. 核心业务流程

## 6.1 B 端主流程

```text
访问官网（unitrace.cn）
  → 注册/申请试用
  → 平台审核或自动试用
  → 选择套餐
  → 网页端支付或线下付款
  → 支付成功自动开通，或超管手动开通
  → 登录租户后台
  → 配置是否开启绑定功能（默认开启）
  → 创建学校/款式/批次
  → 批量创建溯源码
  → 生成微信小程序码
  → 导出 Excel/PDF 标签
  → 印刷/缝制
  → 发放给学生
  → 家长扫码绑定（若租户开启绑定）
  → 数据统计与后续管理
```

---

## 6.2 批量生成溯源码流程

```text
租户员工创建批次
  → 选择学校、款式、尺码、数量
  → 校验订阅状态和剩余码量
  → Redis 预扣码量
  → 创建 generation_task
  → 生成 trace_code 记录（status = pending_qr）
  → 推入 BullMQ 队列
  → Worker 调用微信 wxacode.getUnlimited
  → 图片上传 MinIO
  → 写入 wxacode_asset
  → 更新 generation_task 成功/失败数量
  → trace_code.status = unbound
  → 失败进入重试队列或死信队列
```

前置条件：

```text
只有 status = pending_qr 的溯源码可提交生成。
生成成功后状态变为 unbound。
生成失败后状态保持 pending_qr，可重试。
```

---

## 6.3 C 端扫码总流程

```text
用户微信扫码
  → 微信识别小程序码
  → 打开小程序「衣码归」pages/trace/detail
  → 解析 scene
  → 请求后端详情接口（无需登录即可获取物品信息）
  → 后端判断：租户是否开启绑定、码状态、访问者是否为绑定人
  → 返回差异化数据
  → 返回 traceToken（仅未绑定且租户开启绑定时）
  → 小程序展示页面
```

说明：

```text
物品基础信息（学校、款式、尺码、颜色、批次、状态）无需登录即可访问。
绑定操作需要微信登录和 traceToken。
挂失状态下任何人都能看到醒目的「已丢失」+ 学校 + 班级 + 归还指引。
租户关闭绑定时，扫码仅展示物品信息，不返回 traceToken，不出现登录/绑定入口。
未登录用户通过 IP + 设备指纹进行限流。
已登录用户通过 OpenID 进行限流。
```

---

## 6.4 小程序内扫码流程

```text
用户进入小程序「衣码归」
  → 点击"扫一扫"
  → 调用 wx.scanCode
  → 扫描校服上的微信小程序码
  → 解析扫码结果
      1. 从 path 中解析 scene
      2. 从 result 中解析 scene_code
      3. 从 result 中解析 code_no
  → 校验 code_no 校验位
  → 跳转详情页
  → 如果解析失败
      → 提示手动输入编号
      → 手动输入触发限流
```

---

## 6.5 绑定流程（v5.0）

```text
扫码
  → 租户已开启绑定 且 状态 unbound
  → 点击立即绑定
  → 微信登录（wx.login + code2Session）
  → 使用详情接口返回的 traceToken
  → 填写「班级」（学校来自溯源码，不可改）
  → 勾选用户协议
  → 提交绑定
  → 后端事务校验
  → 校验 traceToken 有效性和作用域
  → 校验租户绑定开关为开启
  → 插入 binding（仅存 class_name + c_user_id）
  → 更新 trace_code.status = bound
  → 绑定成功
```

> 不再有手机号填写、短信验证码、披露授权询问等环节。

租户关闭绑定时：扫码不进入此流程，仅展示物品信息。

---

## 6.6 挂失流程（v5.0：纯展示）

```text
家长进入我的校服
```

### 6.8.4 支付掉单补偿流程

```text

  → 后端主动调用支付渠道查询订单
  → 若渠道已支付，则本地补单
  → 更新订单 paid
  → 开通订阅
  → 记录异常补偿日志
```

定时补偿任务：

```text
BullMQ 定时任务每 5 分钟扫描 status = paying 且超过 5 分钟的订单。
主动查询支付渠道状态。
如果渠道已支付则补单。
如果渠道未支付且超过订单有效期则标记 expired。
```

# 7. 状态机设计

## 7.1 溯源码状态机

```text
pending_qr
    │
    │ 提交生成小程序码
    ▼
generating
    │
    │ 生成成功
    ▼
unbound
    │
    │ 家长绑定
    ▼
bound
    │
    │ 挂失
    ▼
```

## 9.8 解析扫码结果

```ts
type ScanParseResult =
  | { type: 'scene'; value: string }
  | { type: 'codeNo'; value: string };

function parseScanResult(res: any): ScanParseResult | null {
  // 微信小程序码扫码结果中 path 包含 scene 参数
  if (res.path) {
    const match = res.path.match(/[?&]scene=([^&]+)/);
    if (match && match[1]) {
      return {
        type: 'scene',
        value: decodeURIComponent(match[1])
      };
    }
  }

  if (res.result) {
    const result = decodeURIComponent(res.result).trim();

    // 匹配 scene_code：8-12 位字母数字
    if (/^[A-Za-z0-9]{8,12}$/.test(result)) {
      return {
        type: 'scene',
        value: result
      };
    }

    // 匹配 code_no：UT-XXXX-XXXX 格式
    if (/^UT-[0-9A-HJKMNP-TV-Z]{4}-[0-9A-HJKMNP-TV-Z]{4}$/i.test(result)) {
      const codeNo = result.toUpperCase();
      if (validateCodeNoChecksum(codeNo)) {
        return {
          type: 'codeNo',
          value: codeNo
        };
      }
    }
  }

  return null;
}
```

## 12.3 鉴权方式

```text
B 端：账号密码登录 + JWT access_token + refresh_token
C 端：微信登录 + JWT access_token
敏感操作：短期 traceToken
支付回调：签名验证，不依赖用户登录态
```

接口访问级别：

```text
公开接口：扫码详情物品信息（无需登录，IP 限流）
C 端登录接口：微信登录（无需 token）
C 端认证接口：绑定、挂失、解绑（需 JWT + traceToken）
B 端认证接口：后台管理（需 JWT）
超管接口：平台管理（需超管 JWT）
回调接口：支付回调（签名验证，无需 JWT）
```

建议：

```text
access_token 有效期：2 小时
refresh_token 有效期：14 天
refresh_token 轮换
```

## 12.11 支付回调接口

### 微信支付回调

```http

```

### 支付宝回调

```http
POST /api/payments/notify/alipay
```

处理规则：

```text
1. 验签。
2. 校验商户号。
3. 校验订单金额。
4. 幂等处理（同一 transaction_id 不重复处理）。
5. 更新 payment。
6. 更新 order。
7. 开通订阅。
8. 返回成功应答。
```

安全要求：

```text
回调接口不依赖用户 JWT。
回调接口通过签名验证确认来源。
微信支付回调来源 IP 可选白名单校验。
回调接口必须防重放（时间戳 + nonce）。
```

# 14. 安全、隐私与合规

## 14.1 数据分类

| 数据         | 敏感级别 | 处理方式                   |
| ------------ | -------- | -------------------------- |
| 微信 OpenID  | 中       | 身份识别（假名）           |
| 班级         | 低       | 按需展示（挂失时）         |
| 校服基础信息 | 低       | 可公开                     |
| 申诉附件     | 中       | 私有桶签名 URL             |
| 支付凭证     | 高       | 私有桶签名 URL             |
| 支付密钥     | 极高     | 环境变量/KMS，禁止入库明文 |
| B 端企业联系人电话 | 中 | B 端企业信息，非 C 端个人隐私 |

> v5.0：系统不存储学生姓名、家长电话等 C 端个人身份信息，因此数据分类大幅简化，未成年人个人信息保护压力显著降低。

---

## 14.2 加密规则

| 数据       | 加密方式                                 |
| ---------- | ---------------------------------------- |
| B 端/平台用户密码 | argon2id 或 bcrypt                       |
| 文件 URL   | 临时签名                                 |
| 数据库备份 | 加密存储                                 |
| 支付密钥   | 环境变量/KMS/配置中心                    |
| traceToken | HMAC-SHA256 签名，内含加密 trace_code_id |

---

## 14.3 密钥管理

```text
支付商户密钥不得写入代码仓库。
生产环境使用环境变量或 KMS。
支持密钥轮换。
不同环境使用不同密钥。
备份文件加密。
导出文件签名 URL 有效期不超过 1 小时。
支付私钥文件权限严格控制（chmod 600）。
```

---

## 14.4 留痕规则

所有支付相关操作记录：

```text
谁创建订单；
订单金额；
支付渠道；
支付时间；
回调时间；
回调验签结果；
是否开通订阅；
是否退款；
退款审批人；
退款时间。
```

所有绑定/挂失/解绑操作记录操作日志（操作人 OpenID、时间、对象、结果）。

---

## 14.5 日志脱敏

```text
request_payload 不得记录密码。
不得记录支付密钥。
不得记录完整银行卡号。
敏感字段进入日志前必须脱敏。
```

> v5.0：由于不存储手机号、学生姓名、短信验证码，日志脱敏规则相应简化。

---

## 14.6 合规要求

```text
1. 官网和小程序必须提供隐私政策。
2. 小程序必须配置微信隐私保护指引。
3. 个人信息收集最小化：C 端只收集班级（集体信息），不收集学生姓名、家长电话等个人身份信息。
4. 提供账号注销入口。
5. 提供数据更正、删除、撤回同意入口。
6. 法律要求保留的审计日志可匿名化保留。
7. 发生安全事件应有应急预案。
8. 域名需完成 ICP 备案和公安备案。
9. 网页支付启用前需完成支付商户资质配置。
10. 支付页面需展示退款规则和发票说明。
```

> v5.0：因不收集未成年人个人身份信息（只存学校与班级这类集体信息），未成年人个人信息保护的合规义务大幅减轻，但仍需在隐私政策中如实说明收集范围。

# 15. 风控与防刷

## 15.1 限流规则

| 操作               | 限制                     |
| ------------------ | ------------------------ |
| 扫码详情（已登录） | 同一 OpenID 每分钟 30 次 |
| 扫码详情（未登录） | 同一 IP 每分钟 20 次     |
| 编号查询（已登录） | 同一 OpenID 每分钟 10 次 |
| 编号查询（未登录） | 同一 IP 每分钟 5 次      |
| 绑定提交           | 同一 OpenID 每分钟 3 次  |
| 导出任务           | 租户每分钟 5 次          |
| 申诉提交           | 同一 OpenID 每天 5 次    |
| 创建订单           | 同一租户每分钟 5 次      |
| 支付回调           | 来源 IP 白名单/验签保护  |

说明：

```text
未登录用户通过 IP + 设备指纹（User-Agent 哈希）进行限流。
已登录用户通过 OpenID 进行限流。
触发限流后返回 42901 错误码。
高频编号查询触发临时限流。
```

# 22. 最终开发基线总结

```text
1. 项目名称：衣码归（UniTrace）。
2. 项目核心是不重复造轮子。
3. 使用微信小程序码，不生成普通二维码。
4. 第三方普通扫码工具不能直接解析为业务入口，但不承诺绝对不可识别。
5. 标签包含小程序码、编号（UT 前缀）、学校、款式、尺码、颜色、批次、生产日期。
6. 小程序「衣码归」内提供扫一扫，使用 wx.scanCode。
7. 小程序内扫码失败时提供编号输入兜底。
8. code_no 必须随机并带校验位（Crockford Base32 + Luhn mod 32），前缀 UT。
9. scene_code 必须随机，不能使用 Base62(trace_code.id)。
10. 绑定接口不直接暴露 trace_code.id，使用 traceToken。
11. traceToken 为短期签名 token，有效期 5～10 分钟，绑定成功后绑定用途失效。
12. 任何人可绑定，但一件校服只能绑定一次。
13. 绑定只需微信登录 + 填写班级，不验证电话，不发短信。
14. 系统只保存学校与班级，不保存学生姓名、家长电话等个人身份信息。
15. 已绑定后其他人不能继续绑定。
16. 非绑定人不可查看绑定人信息。
17. 挂失功能唯一作用：扫码时醒目显示「已丢失」+ 学校 + 班级 + 归还指引。
18. 挂失不通知家长（家长是发起方）。
19. 绑定功能可由租户开关，关闭后扫码退化为普通电子标签（无登录/绑定/挂失）。
20. 订阅到期后 C 端扫码查询继续有效。
21. 租户到期后 B 端禁止新增、生成、导出、批量操作。
22. 当前无小程序资质，开发可先进行。
23. 正式印刷前必须完成正式小程序认证。
24. 正式印刷必须使用 release 版小程序码。
25. 微信小程序内不接入支付。
26. 订阅付费全部在网页端完成。
27. 支付接口必须预留。
28. 有支付资质时，填入支付渠道 Key/商户号/证书即可启用。
29. 未配置支付参数时，使用线下收款和手动开通。
30. 支付配置不是 Cookie，而是商户密钥、API Key、证书、私钥、公钥等。
31. 线上支付成功后自动开通订阅。
32. 线下收款由超管确认后手动开通。
33. 所有订单、支付、退款、订阅变更必须留痕。
34. 技术栈统一为 React + NestJS + TypeScript + PostgreSQL + Redis + MinIO。
35. 小程序使用原生微信小程序 + TypeScript。
36. 初期不拆分开源版和商业版，所有核心功能统一实现。
37. 开源许可证发布前最终确认，若防止闭源商用推荐 AGPL-3.0。
38. 单机 Docker Compose 可运行。
39. API/Worker 无状态，后续可扩展到多台服务器。
40. 必须提供申诉、黑名单、注销、数据删除/匿名化和审计能力。
41. 未登录用户通过 IP + 设备指纹限流，已登录用户通过 OpenID 限流。
42. 定时任务使用 BullMQ repeatable jobs 实现。
43. 账号注销前必须处理活跃绑定。
44. 邮件通知为后续可选，初期不实现。
45. 通知仅用于 B 端站内，C 端家长不做任何通知。
46. 不引入短信服务，不引入行为/图形验证码。
47. GitHub 仓库名：unitrace。npm 包前缀：@unitrace/。数据库名：unitrace。
```

## 23.2 配置说明

```text
PAYMENT_ENABLED=false 时，所有在线支付不可用。
PAYMENT_OFFLINE_ENABLED=true 时，线下收款和手动开通可用。
有支付资质后，将对应渠道配置填入，并开启对应开关。
不填支付参数时，系统不报错，只展示线下收款。
支付参数填写后，系统自动启用对应支付渠道。
BINDING_ENABLED_DEFAULT 控制新租户默认是否开启绑定；
每个租户可在租户配置中独立开关。
```

---

## 23.3 最终执行口径

```text
支付功能先做接口和流程。
有没有资质不影响开发。
有资质就填配置启用。
没资质就线下收款、手动开通。
后续拿到资质后，只需要补配置，不需要重构。

绑定功能默认开启，租户可关闭。
关闭即普通电子标签，零个人信息、零合规风险。
```
